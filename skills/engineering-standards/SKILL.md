# 工程规范

先证明当前系统的真实行为，再决定是否改代码。证据来自当前源码、配置、测试和运行结果。

## 证据门禁

对非简单行为变更，动代码前在内部确认：

1. 现有机制：入口、调用链、数据变化、持久化、读取和展示路径。
2. 缺口证据：已经证明的断点及对应源码位置。
3. 改动边界：在现有所有权模型中负责修复该问题的最小边界。

不强制向用户输出固定模板。根据任务复杂度，用简短进度或审计结论说明关键证据。简单文案、格式或明确的机械修改只需检查受影响文件并直接完成。

在现有链路被追完并证明不足之前，不引入新的机制、分支或旁路绕开问题。

## 审计流程

以当前仓库的实时源码为唯一事实来源。当前链路清楚后，才使用记忆或参考项目补充判断。

1. 确认用户讨论的数据或行为。
2. 追踪它在哪里产生、转换、持久化、加载和展示。
3. 同时检查主动链路和被动链路。运行时行为分别确认触发条件、执行路径、持久化、读取和展示。
4. 分类问题所在边界：
   - 展示层
   - 接口契约
   - 持久化层
   - 运行时行为
   - 外部依赖
   - 异步或周期任务
5. 说明支持分类的证据，以及仍未证明的部分。
6. 排查中发现旁支问题时先记录证据。确认属于当前范围的，修复后再继续测试；与任务无关的只在结尾报告，不擅自扩大改动。只有无法从用户目标判断范围时才询问。

存在 CodeGraph 时可先用它定位，再回到源码和测试验证。不要把索引摘要、记忆或参考项目当成强于当前源码的证据。

## 参考项目

参考项目只用于理解可能的模式。除非同时满足以下条件，否则不把参考实现搬进当前项目：

1. 当前项目确实缺少所需行为。
2. 当前架构存在匹配的所有权边界。
3. 新设计不会绕开或复制现有链路。
4. 当前任务范围允许实现。

参考项目与当前源码冲突时，以当前源码为准。

## 干净实现

优先在现有所有权边界内完成直接修复。

- 除非用户明确要求，不添加兼容代码。
- 不用补丁式分支、重复机制或外层绕路掩盖边界问题。
- 不用大范围重构掩盖不确定性。
- 不为局部问题修改无关模块。
- 只有确实降低复杂度或符合现有模式时才添加抽象。
- 必要重构必须收敛职责、减少分支并让主链路更清楚。
- 未证明现有设计不足时，不引入第二条执行路径。

方案改变行为语义时，动代码前明确说明影响。

## 范围纪律

- 简单问题先给结论。
- 审计给出具体源码证据。
- 不做推测性断言，不虚构事实、数据来源或系统行为。
- 新项目和新链路不保留无意义兼容路径。
- 仓库约定测试目录时，不把测试写入源码模块。
- 用户限定为审计、文档、评审或方案时，只交付对应产物。
- 用户要求直接修复且范围清楚时，不用重复确认阻断执行。

## 动代码前

非简单行为变更应满足：

1. 当前链路已经端到端追完。
2. 问题已按证据分类。
3. 已证明现有机制不能满足需求，而不是尚未找到。
4. 改动落在最小的干净所有权边界。
5. 测试通过公开接口或集成测试面覆盖行为。

缺少任一项时继续调查。简单机械修改不需要套用完整审计流程，但仍要检查目标文件、相邻约束和最终差异。

## 收尾

结束前确认：

1. 运行格式化和相关测试。
2. 运行项目既定 lint。
3. 没有把测试加入错误目录。
4. 没有用参考项目取代当前项目证据。
5. 没有未经批准引入额外分支、旁路或兼容实现。
6. 只有后续判断依赖已改动索引且工具要求手动刷新时，才同步 CodeGraph。
7. 汇报改动、验证结果和明确未做的事项。

## 禁止的做法

1. 只看局部触发点就断言完整系统行为。
2. 没追完已有链路就新增另一套机制。
3. 没证明现有机制失败就设计旁路。
4. 直接照搬参考项目，跳过当前源码证据。
5. 用兼容 shim、重复 API 或生命周期补丁掩盖所有权问题。

正确顺序是：追清现有链路，证明断点，选择所有权边界，完成干净修复，再用真实测试验证。
