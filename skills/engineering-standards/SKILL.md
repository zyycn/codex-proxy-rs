# 工程规范

## 证据顺序

1. 当前运行结果与可复现输入。
2. 当前源码、配置、数据库和日志。
3. 当前测试与公开契约。
4. 架构文档。
5. 记忆和参考项目。

后两项只能辅助定位，不能覆盖当前事实。

## 工作流

1. 追踪入口、调用链、状态变化、持久化、读取和输出。
2. 区分展示、接口、领域规则、存储、异步任务和外部依赖。
3. 找到现有 owner，证明断点位于该边界。
4. 在 owner 内直接修复；必要重构必须减少分支或重复职责。
5. 通过公开接口或外置集成测试固定行为。
6. 运行项目既定格式、lint 和相关测试，再检查最终 diff。

简单文案和机械修改直接完成，不套审计模板。结论先行，只报告影响判断的证据。

## 实现约束

- 不添加兼容 shim、重复 API、双状态机或临时旁路。
- 不用大范围重构掩盖尚未理解的问题。
- 不为局部问题修改无关模块。
- 不在没有复用或所有权收益时增加抽象。
- 用户只要求审计、文档或方案时，不扩大到实现。
- 已确认且属于当前范围的问题，先修复再继续测试。
- 测试遵守仓库目录边界，不混入生产源码。
- 行为语义变化应在修改前明确说明。

## 参考项目

只有同时满足以下条件才采用参考实现：

1. 当前项目已证明缺少该行为。
2. 当前架构存在匹配 owner。
3. 不会复制现有链路。
4. 用户授权的范围包含实现。

参考代码用于验证机制，不用于替代当前项目证据。

## 收尾

交付只包含：

- 已改变的行为与边界。
- 能证明结果的验证。
- 明确未做或仍受外部状态阻塞的事项。

禁止把排查时间线、空洞原则或未证实猜测包装成深度。
