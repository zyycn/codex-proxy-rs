# Codex Proxy RS 项目审计结论

审计日期：2026-07-03

## 结论

项目整体工程质量高于普通自托管代理项目：Rust 后端边界清晰，热路径有较完整的集成测试，账号调度、限流、WebSocket fallback、SSE 终止补偿、使用记录裁剪等关键路径都有明确实现和测试支撑。首轮已完成弱默认密码、管理 Cookie、请求头边界、登录失败节流、账号导出空 ids 拦截和 Docker 默认部署面的加固。当前主要风险不在“代码不可运行”，而在敏感数据落盘形态、运行时任务生命周期、数据库升级机制和发布供应链。

上线前优先处理四类事项：

1. 安全默认值：弱默认管理员密码、管理会话 Cookie `Secure`、登录失败节流、登录审计落库、trusted proxy 显式配置、Docker 本机绑定、容器 healthcheck/rootless/security options 已完成首轮整改；管理端 CSP 已移除 `unsafe-inline`。
2. 敏感数据：客户端 API Key、admin API Key、账号 access/refresh token、Cloudflare cookie 目前均明文存 SQLite，需要分层加密或不可逆哈希策略；该项按本轮决策暂缓，不与当前加固批次混做。
3. 运维闭环：WebSocket 池已纳入 `TaskCoordinator` 统一 shutdown；构建与安全扫描已有 CI；本轮已清理当前 RustSec warning 来源。
4. 在线更新供应链：系统更新接口有管理员鉴权、进程内锁、文件锁、下载大小上限、GitHub API base 信任边界、HTTPS/GitHub host 校验、强制 `checksums.txt` SHA256 校验和 tar 路径穿越检查；发布 workflow 已增加 release asset provenance、容器镜像 SBOM/provenance 和 cosign keyless 签名，运行时更新客户端仍需加入签名/provenance 验证。

## 已有强项

- 后端启用 `unsafe_code = "forbid"`，clippy 基线较严格，且当前全量 clippy 通过。
- 代理调度路径对 quota exhausted、rate limit、expired、disabled、banned、Cloudflare challenge/path block、model unsupported 做了细分类和 fallback。
- 流式响应路径在客户端取消、上游错误、正常完成时都释放账号槽位，并补齐 `response.failed` / `[DONE]`，降低悬挂连接和 OpenAI 协议不完整风险。
- 使用记录默认不保存请求/响应体，只记录必要元数据；容量默认裁剪，避免无限增长。
- SQLite schema 有较多约束和索引，测试覆盖了当前表结构、非法状态、负计数、运行时设置等。
- 管理端静态资源带 CSP、`nosniff`、`x-frame-options`、`referrer-policy`，且 CSP 已限制 script/style 到 `self`，不再允许 `unsafe-inline`。
- 管理端 session Cookie 现在包含 `Secure; HttpOnly; SameSite=Lax`。
- 启动时会拒绝空密码、过短密码和常见弱默认管理员密码；Docker 示例配置不再提供可用弱密码。
- CI 已覆盖 Rust fmt/clippy/test、前端 format/build、Compose config 和 runtime Docker 镜像构建；单独 security workflow 覆盖 cargo audit、pnpm audit 和 runtime 镜像 Trivy 扫描；当前 RustSec warning 已清零；workflow action 引用已 pin 到 commit SHA。
- Docker 运行阶段已改为非 root 用户，Compose 默认仅绑定 `127.0.0.1:8080`，并包含 healthcheck、`no-new-privileges`、cap drop、`init`、停止宽限、pid/CPU/内存边界、`/tmp` tmpfs 和命名卷。
- release workflow 会产出多平台二进制、Docker 镜像和 `checksums.txt`；release archives 会生成 GitHub artifact attestation，Docker 镜像会随 BuildKit 发布 SBOM/provenance attestations 并用 cosign keyless 签名；缺口转为运行时更新客户端验证这些证明。
- 系统更新入口均要求 `require_admin_auth`；更新和回滚同时使用进程内 `SYSTEM_OPERATION_LOCK` 与文件锁，能防止常规并发更新。
- 更新包处理有 500 MiB 下载上限、HTTPS/GitHub 下载域名白名单、tar 路径穿越拦截、跨文件系统移动 fallback、web/binary 备份与失败回滚。
- 前端更新弹窗把 release notes 当文本渲染，没有用不可信 HTML；SSE/EventSource 使用 cookie session，并在弹窗关闭或组件卸载时断开。

## 关键证据索引

- `backend/src/config/types.rs`：`AdminConfig::default()` 的 `default_password` 为空；`validate_default_password()` 拒绝空值、短密码和常见弱口令。
- `backend/src/config/loader.rs` / `backend/src/runtime/bootstrap.rs`：启动会优先读取 `CPR_CONFIG_FILE` 指向的配置文件，未设置时回退到当前目录 `config.yaml`；在没有管理员时用 `default_password` 创建默认管理员，创建前会先校验默认密码。
- `backend/src/admin/auth/session.rs`：登录和登出 Cookie 字符串包含 `Secure; HttpOnly; SameSite=Lax`；登录成功、凭证失败、限流和内部错误会写入 `admin_auth` 审计记录，元数据只保留来源和是否提供用户名，不写入用户名、密码或 session id。
- `backend/src/admin/auth/service.rs`：同一来源管理员登录失败达到 5 次后短期锁定，成功登录会清理该来源失败计数。
- `backend/src/proxy/auth.rs`：`/v1` client API key 鉴权会区分 `missing_authorization`、`malformed_authorization`、`invalid_key_format`、`invalid_key` 和 `store_unavailable`，并写入结构化日志；对外仍返回统一 OpenAI `invalid_api_key`。
- `backend/src/admin/keys/service.rs`：`AdminClientApiKey` 含完整 `key` 字段，SQLite `create` 直接写入完整 key，`list()` / `get()` 会把存储对象映射给管理端返回；运行时 cache 也以明文 key 为查找键。
- `backend/src/admin/keys/routes.rs`：`ClientApiKeyData` 响应结构包含完整 `key`。
- `backend/tests/admin/keys/store.rs`：测试显式断言数据库 `key` 字段和列表返回值等于创建出的完整 key，说明这是当前行为契约而不是偶然泄露。
- `frontend/src/views/api-keys/composables/useApiKeyMutations.ts`：UI 会用 prefix 做遮罩展示，但后端响应、网络传输和前端内存里仍可能存在完整 key。
- `backend/src/config/settings.rs`：`runtime_settings.admin_api_key` 保存完整 admin API key；校验用 constant-time compare，展示状态时才 masking。
- `backend/src/admin/accounts/routes.rs` / `backend/src/admin/accounts/service/exporting.rs`：导出接口要求 `confirm=export_sensitive_accounts` 且 ids 非空；成功、缺少确认、空 ids、账号不存在和内部失败会写入 `admin_account_export` 审计记录；按 ids 导出时 `ExportedAccount::from` 仍会明文导出 access token 和 refresh token。
- `backend/src/http/middleware/request_id.rs`：只接受长度不超过 128 且由字母、数字、`_`、`-`、`.` 组成的 `x-request-id`；`TrustedProxyConfig` 支持 IP/CIDR。未命中 trusted proxy 时会移除 `cf-connecting-ip`、`x-real-ip`、`x-forwarded-for` 并写入 TCP peer IP；命中 trusted proxy 时才从转发头提取客户端 IP，并统一折叠为规范化 `x-real-ip`。
- `backend/src/web/headers.rs`：静态资源 CSP 当前为 `script-src 'self'` 和 `style-src 'self'`，已删除长期 `unsafe-inline` 依赖。
- `backend/src/http/middleware/trace.rs`：日志 span 和请求日志保留 URI path 与 query 参数名、顺序和重复结构，但所有参数值均写成 `<redacted>`；无 `=` 的裸 query 段也会整体脱敏。
- `backend/src/infra/database.rs` / `backend/src/infra/migrations/0001_initial.sql`：SQLite 初始化启用 WAL、foreign keys、busy timeout，并通过 `schema_migrations` 执行编号迁移；业务表 DDL 不再使用 `create table if not exists` 作为隐式升级机制。
- `backend/src/admin/system/routes.rs`：`perform_update` 有管理员鉴权、mutex、文件锁、GitHub API base 校验、下载域名校验、强制 HTTPS 下载和强制 `checksums.txt` 校验；`CPR_UPDATE_ALLOW_INSECURE_DOWNLOADS=true` 不再放行非本地同源 HTTP 下载；checksum 校验仍来自同一 release；`restart` 现在请求运行时优雅关闭信号，不再直接 `std::process::exit(0)`。
- `backend/src/http/router.rs`：`/healthz` 会执行 SQLite `select 1`，数据库不可用时返回 503。
- `backend/src/admin/monitoring/usage_record_store.rs`：`capture_body=false` 时会移除请求/响应 body 字段，这是正确默认；`record_audit()` 用于安全审计事件，不受普通 usage logging 开关影响，但仍沿用容量裁剪和 body 捕获策略。
- `deploy/Dockerfile`：runtime 阶段创建并使用 `cpr` 非 root 用户，内置 `/healthz` healthcheck。
- `deploy/docker-compose.yml`：示例端口为 `127.0.0.1:8080:8080`，并配置 healthcheck、`no-new-privileges`、cap drop、非 root 用户和命名卷。
- `deploy/Dockerfile.dockerignore`：Dockerfile 专用 ignore 已覆盖 build output、依赖缓存、本地配置、运行数据、工具状态，并保留 `dist/docker` 给 prebuilt runtime target 使用。
- `.github/workflows/ci.yml` / `.github/workflows/security-scan.yml` / `.github/workflows/release.yml`：CI 已加入前端格式检查、Compose config、runtime 镜像构建和 Trivy 镜像扫描；workflow action 均已 pin 到 commit SHA；release workflow 的写权限已收敛到 publish job。
- `.github/workflows/release.yml`：发布产物有 `checksums.txt`，release archives 会生成 GitHub artifact attestation，Docker 镜像发布启用 BuildKit SBOM/provenance，并通过 cosign keyless 签名；后续仍需让在线更新流程验证这些证明。

## 分层发现

### 认证与会话

- 管理端 session 使用 Argon2 哈希密码、UUID session id、HttpOnly Cookie，基础方向正确。
- Cookie 已包含 `Secure; HttpOnly; SameSite=Lax`；如果经反向代理终止 TLS，应把会覆盖或清理转发头的代理 IP/CIDR 写入 `server.trusted_proxies`。
- 登录接口已加入同来源失败计数和短期锁定，暴力尝试成本已提高；成功、凭证失败、限流和内部错误均会写入 `usage_records` 的 `admin_auth` 审计事件。
- 当前请求入口不再默认信任 `cf-connecting-ip`、`x-real-ip` 和 `x-forwarded-for`；只有 TCP peer 命中 `server.trusted_proxies` 时才采信转发头，否则统一使用 TCP peer IP。
- `x-request-id` 已限制长度和字符集，非法值会被替换为服务端生成的新 request id。
- 管理端 API key 从 `x-api-key` 读取，使用 constant-time 比较，但 key 明文存库。
- `/v1` client API key 校验基于内存表，热路径性能好；鉴权失败原因已结构化记录但对外保持统一 401。明文 key 存库并在管理端元数据中长期可见，安全模型仍偏弱。
- 管理端 client key 列表结构包含完整 `key` 字段，这意味着只要拿到管理端会话就能长期复制所有代理 key；建议只保留创建时一次性明文返回。

### 配置与部署

- `AppConfig` 使用 `deny_unknown_fields`，能阻止拼错字段静默生效。
- 配置加载已支持 `CPR_CONFIG_FILE` 显式路径；未设置时仍回退到当前目录 `config.yaml`，本地直接运行需要注意工作目录。
- `server.trusted_proxies` 默认为空列表，支持单个 IP 和 CIDR；默认会剥离客户端传入的转发 IP 头，反代部署需要显式配置可信 peer。
- Docker Compose 默认暴露 `127.0.0.1:8080:8080`，避免直接监听宿主所有网卡。
- runtime 镜像基于 `debian:bookworm-slim`，安装 CA 和 curl healthcheck，默认以 `cpr` 非 root 用户运行。
- `deploy/Dockerfile.dockerignore` 会收窄 Dockerfile 构建上下文；由于 Dockerfile 位于 `deploy/`，这里使用 Dockerfile 专用 ignore，而不是根 `.dockerignore`。
- Compose 已设置 `healthcheck`、`security_opt: no-new-privileges:true`、capability drop、非 root 用户、命名卷、`init`、停止宽限、pid/CPU/内存边界和 `/tmp` tmpfs；`read_only` 根文件系统仍需结合自更新路径评估。
- runtime 镜像显式创建 `/app/data`、`/app/logs`、`/app/web` 并赋予非 root 用户；默认命名卷会继承容器目录属主。

### 数据库与升级

- SQLite 连接初始化会创建父目录、启用 WAL、foreign keys 和 5 秒 busy timeout，这些默认值适合单进程自托管场景。
- schema 有表约束、索引和运行时设置表，测试也覆盖了当前 schema 的多类约束。
- 当前初始化先确保 `schema_migrations` 元数据表存在，空库按顺序应用 `0001_initial.sql` 并记录版本 1；已有业务表但没有迁移记录的库会拒绝启动，避免把重复执行 schema 当成隐式升级。
- 未来新增列、收紧约束、回填索引或迁移敏感字段加密形态时，需要新增编号 migration 并提升当前版本，否则启动会因未知迁移版本失败。

### 在线更新与发布供应链

- `version`、`check_updates`、`update_event_stream`、`perform_update`、`update_status`、`rollback`、`restart` 均要求管理员认证，这是正确边界。
- `perform_update` 使用进程内 mutex 和文件锁，`OperationLock` 有 30 分钟 stale lock 处理，能处理大部分重复点击和崩溃残留。
- 下载 URL 默认限制 HTTPS 且只允许 GitHub / objects.githubusercontent.com；`CPR_UPDATE_ALLOW_INSECURE_DOWNLOADS=true` 不再绕过 HTTP 拒绝，明显降低任意下载风险。
- `extract_release_archive` 拒绝绝对路径、父目录和平台 prefix，避免 tar 包写出临时目录。
- 当前已要求 release 必须提供 `checksums.txt`，缺失、不匹配或 checksum 下载失败时更新失败；生产 `CPR_GITHUB_API_BASE` 只能指向官方 GitHub API，避免 release 元数据来源被环境变量改到任意 endpoint。checksum 校验仍只是从同一个 release 下载文本并比对 SHA256，不能证明发布者身份；后续应验证 GitHub artifact attestation 或独立签名。
- release workflow 会构建多平台二进制和 GHCR Docker 镜像，并发布 `checksums.txt`；release archives 已接入 GitHub artifact attestation，镜像已启用 SBOM/provenance attestations 和 cosign keyless signing。镜像漏洞扫描已在 security workflow 覆盖，不在 release 发布链路阻断。
- GitHub Actions 依赖已从 tag ref 改为 commit SHA pinning，并保留原 tag 注释便于后续升级审计。
- `CPR_GITHUB_API_BASE` 可由环境变量改写，但生产校验只允许不带 credentials/query/fragment 的 `https://api.github.com/repos`；本地 loopback `/repos` 仅用于测试。后续仍需要签名/provenance 兜底 GitHub release 资产本身的信任问题。
- `replace_release_files` 先替换 web，再备份/替换 binary，并做失败回滚；实现有恢复意识，但不是严格事务，断电/进程 kill 仍可能留下 backup 与当前文件不一致的中间状态。
- `restart` 使用 `tokio::spawn` 延迟 500ms 后请求运行时 shutdown signal，服务端会走 axum graceful shutdown 并在退出前关闭后台任务；仍依赖外部 supervisor 或 Docker restart policy 拉起新进程。

### 代理和并发

- 账号池用 `tokio::Mutex<AccountPool>` 串行化内存状态，逻辑简单可靠；调度计算在锁内，持久化在锁外，避免长时间持锁。
- 流式响应的取消路径释放账号槽位，降低 stale slot 依赖。
- `stale_slot_ttl` 仍是必要兜底，应补充测试覆盖 panic/abort 或 spawn 任务异常退出后的容量恢复。
- WebSocket pool 有 idle maintenance、ping、liveness timeout 和 shutdown 方法；当前 Services 会保留 pool 句柄，`TaskCoordinator` 关闭时会调用 pool shutdown，让后续 acquire 绕过池并关闭 idle 连接。

### 数据与隐私

- 请求/响应体默认不落使用记录，这是正确默认。
- `accounts.access_token`、`accounts.refresh_token`、`account_cookies.value`、`client_api_keys.key`、`runtime_settings.admin_api_key` 目前都是明文。
- SQLite 文件一旦泄露，可直接获得上游账号 token、cookie 和代理 key；这是当前最大实际安全风险。
- 账号导出现在要求显式 ids 和固定确认词，不再允许空 ids 导出全部；成功和失败路径会写入不含 token/email 的 `admin_account_export` 审计记录。但选中账号仍会把 `StoredAccount` 的 `access_token` 和 `refresh_token` 通过 `ExposeSecret` 转成明文 JSON。迁移能力还需要频率限制和可选加密导出包。
- 使用记录默认移除 `body`、`rawBody`、`requestBody`、`responseBody`、`upstreamBody`，这是好的隐私默认；如果开启 capture body，需要在 UI 和配置层明确提示风险，并考虑字段级脱敏。

### 前端管理面

- 路由守卫会在首次进入非登录页时检查 session，基本闭环。
- Axios 设置 `withCredentials: true`，和 cookie session 匹配。
- API 错误对象未导出类型，调用处多用 `any`，前端错误处理和重构安全性一般。
- 登录页和管理页依赖后端 session 状态，不把 token 放 localStorage，这是正确方向。
- 未发现前端使用 `v-html` 或 `innerHTML` 渲染后端文本；release notes、日志和使用记录展示默认走文本插值。
- API key 页面会用 prefix 生成 masked key 展示，这是好的交互；但后端列表响应仍包含完整 key，前端遮罩不能降低网络、浏览器内存、日志或调试工具里的暴露风险。
- 系统更新 EventSource 使用 `withCredentials: true`，后端也要求 session；需要注意 SSE 长连接断线后的错误状态和自动重连策略，避免 UI 显示“断开”但底层浏览器仍在重连。
- `useDownload` 会把 JSON 导出为本地下载，这本身合理；结合账号导出时需要把“下载文件就是完整密钥包”的风险在操作层显式化。

### CI 与供应链

- CI 基线足够覆盖主开发路径。
- `cargo audit` 当前无漏洞或 allowed warnings；后续重点转向重复依赖收敛和 fork 依赖维护策略。
- `Cargo.toml` 使用 forked `tungstenite` / `tokio-tungstenite`，这可能是必要工程选择，但应有版本更新和安全响应责任说明。
- 前端 `pnpm audit --prod --audit-level=high` 通过；构建警告应尽快消除，避免真实构建问题被噪音淹没。
- security workflow 已跑 cargo audit、pnpm audit 和 runtime 镜像 Trivy 扫描；GitHub Action 依赖已 pin 到 commit SHA；release asset provenance 和镜像签名已在发布链路配置，客户端验证仍未接入。
- release workflow 默认只有 `contents: read`，只有 package-assets job 额外拥有 `id-token: write`/`attestations: write` 用于 release asset attestation，publish job 拥有 `contents: write`、`packages: write`、`id-token: write` 和 `attestations: write` 用于发版、镜像发布、provenance 与 cosign keyless signing。
